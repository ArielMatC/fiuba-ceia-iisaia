# Oil Production Forecaster — TP Final (IISAIA / FIUBA)

Web app that forecasts per-well oil production using Arps decline curves. Backend: FastAPI + SQLAlchemy 2.x + SQLite + SciPy. Frontend: React 19 + TypeScript + Vite.

Current state: MVP complete (Phases 0–4). Backend with 31 tests at ~95% coverage; typed React SPA; verified end-to-end in a browser.

## Source of truth

  • ARCHITECTURE.md — structural map (modules, data model, API, flows)
  • STYLEGUIDE.md — brand identity, color/typography tokens, UI patterns
  • docs/decisions.md — design decision log (ADR-lite, the why)
  • tasks/todo.md — phase plan + per-phase Review (tracker, not design)
  • tasks/lessons.md — lessons across sessions (injected by a hook)
  • docs/presentation.html — project presentation (scope)

## Commands

Backend (from backend/, managed with uv — never call python directly, the system default is 3.9):

  • uv sync — create/update the env (Python 3.12, pinned)
  • uv run uvicorn app.main:app --reload --port 8000
  • uv run pytest --cov=app — tests / coverage
  • uv run ruff check . / black --check .
  • uv run python -m scripts.export_openapi — export spec (enforced by a sync test)

Frontend (from frontend/): npm run dev · build · lint · format.

## Workflow

  • Each phase starts in plan mode and closes with /phase-review
  • Done only when: tests pass, coverage ≥ 85%, demo verifiable, Review updated
  • Detailed conventions live in .claude/rules/

## Project Learning Protocol

  1. After any correction, decision, or gotcha → run /learn
  2. Routing: lessons → lessons.md; decisions → decisions.md; stable facts → this file or .claude/rules/
  3. Keep this file short: stable facts only, no history
  4. End any session that changed code by running /learn

# Architecture — Oil Production Forecaster

Structural map of the system: what each part is, where it lives, and how a request flows. The why lives in docs/decisions.md; phase status in tasks/todo.md; UI tokens in STYLEGUIDE.md.

Maintenance: update in the same change that alters structure — new module, table, endpoint, page, or a moved responsibility. See .claude/rules/docs.md.

## 1. Overview

A single-user web app that forecasts per-well oil production with Arps decline curves.

┌──────────────┐    HTTP/JSON     ┌──────────────┐   SQLAlchemy   ┌──────────┐
│  Frontend    │ ───────────────▶ │   Backend    │ ─────────────▶ │  SQLite  │
│ React 19 SPA │ ◀─────────────── │   FastAPI    │ ◀───────────── │  (file)  │
│ (port 5173)  │      CORS        │ (port 8000)  │                │          │
└──────────────┘                  └──────────────┘                └──────────┘

No auth, single SQLite file, schema created at startup. CORS restricted to http://localhost:5173.

## 2. Backend (backend/)

Strict layering — routers do HTTP only, services hold business logic, models are the ORM.

LayerPathResponsibility
App / lifespanapp/main.pyFastAPI app, CORS, router wiring, init_db() on startup, GET /api/health
DBapp/db.pyEngine, SessionLocal, get_db dependency, init_db
Models (ORM)app/models/Well, ProductionData, Forecast, ForecastData
Schemasapp/schemas/Pydantic v2 request/response models
Routersapp/api/*.pyHTTP endpoints, validation, status codes
Servicesservices/csv_loaderPure CSV parse + validation (no DB)
services/wellsUpsert/ingest, aggregates, production queries
services/arpsDecline-curve fit / predict / EUR (pure math)
services/forecastOrchestrates fit + persists forecast series
Scriptsscripts/export_openapiDumps the live schema to docs/openapi.yaml

### Request flows

Upload (POST /api/wells/upload): router reads the file → csv_loader.parse_production_csv (bytes → validated records + rejected rows; missing columns → 422) → wells.ingest_records (upsert well by name, upsert production by (well_id, date), last-wins) → UploadSummary.

Forecast (POST /api/wells/{id}/forecast): router validates model_type → forecast.run_forecast loads production, converts dates to months-since-first (float), arps.fit (bounded curve_fit, hyperbolic falls back to exponential), computes a month limit, caps arps.eur at the economic cutoff and persists monthly points only while rate ≥ cutoff → ForecastDetail.

## 3. Data model (SQLite)

  • wells — id, name (unique), field, operator, oil_rate_unit, created_at
  • production_data — id, well_id (fk, cascade), date, oil_rate; unique (well_id, date)
  • forecasts — id, well_id (fk, cascade), model_type, params_json, r_squared, horizon_months, economic_cutoff, eur, created_at
  • forecast_data — id, forecast_id (fk, cascade), date, predicted_rate

All deletes cascade. No migrations (MVP) — recreate the DB file on schema change.

## 4. API surface

Spec auto-generated by FastAPI (/docs, /redoc) and exported to docs/openapi.yaml; test_openapi_sync.py fails if it drifts.

## 5. Domain — Arps decline curves

Three variants by exponent b: exponential (b=0), hyperbolic (0<b<1), harmonic (b=1). Fit with scipy.optimize.curve_fit under physical bounds; time is months-since-first-point (float, no resampling). EUR integrates the fitted rate to the economic-cutoff time or the displayed limit, then scales months→days. Formulas in .claude/rules/domain-arps.md.

## 6. Frontend (frontend/)

React 19 + TypeScript (strict) + Vite. State/fetching via TanStack Query; charts via Recharts; styling via Tailwind v4. Visual tokens in STYLEGUIDE.md.

Routes: / (welcome/login), /wells, /upload, /wells/:wellId.

## 7. Cross-cutting

  • Config: backend port 8000; frontend VITE_API_URL (default http://localhost:8000)
  • CORS: backend allows http://localhost:5173 only
  • Toolchain: backend via uv (Python 3.12; system python is 3.9 — never call it); frontend via npm
  • Tests: backend pytest (~95% coverage); frontend lint + build

## 8. Risks & constraints (MVP)

  • Hyperbolic convergence: noisy/short series can defeat curve_fit; mitigated by bounds, initial guess, exponential fallback
  • Irregular dates: time is months-since-first as float; no resampling
  • EUR ↔ cutoff: comparable only at equal cutoffs/horizons; cutoff always stored
  • Single scale: only oil_rate is forecast; gas/water out of scope
  • No migrations / no auth / single user: deliberate MVP scope
